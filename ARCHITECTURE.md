# Next.js Landing Page Architecture Documentation

This document outlines the standard architecture and best practices for developing and maintaining the Apex Web Studios landing page. It is structured to ensure scalability, performance, SEO optimization, and an excellent developer experience.

## 1. Project Structure

The project follows a strict separation of concerns, grouped by feature and architectural layer.

```text
├── app/                  # Next.js App Router (Routing & Pages)
│   ├── page.js           # Landing page root (Server Component)
│   ├── layout.js         # Root layout (Metadata, html/body, providers)
│   ├── loading.js        # Suspense UI (Optional)
│   ├── error.js          # Error boundary (Optional)
│   ├── not-found.js      # 404 page (Optional)
│   ├── sitemap.js        # Auto sitemap generation
│   └── robots.js         # robots.txt generation
├── components/           # React Components
│   ├── ui/               # Atoms & generic reusable components (Button, Input, Accordion)
│   ├── sections/         # Landing Page sections (Hero, Features, Services, etc.)
│   ├── layout/           # Shared layout wrappers (Navbar, Footer)
│   ├── seo/              # SEO components (JSON-LD, OpenGraph setups)
│   ├── icons/            # SVG icons
│   └── providers/        # Context wrappers
├── lib/                  # Utility functions and helpers
│   ├── utils.js          # Helper functions (e.g., clsx, tailwind-merge)
│   └── fonts.js          # next/font configurations
├── config/               # Global configuration variables
│   └── site.js           # Site metadata, navigation links, constants
├── public/               # Static Assets
│   └── images/           # Images, SVGs, favicon
├── styles/               # Global styles
│   └── globals.css       # Tailwind base, utilities, and CSS variables
├── next.config.mjs       # Full Next.js configuration
└── middleware.js         # Edge logic and request interception (Optional)
```

## 2. Rendering Strategy & Loading
- **Server Components by Default**: All components in `app/` and `components/sections/` should remain Server Components unless they require state or interactivity.
- **Client Components**: Only use `"use client"` directive at the top of the file for interactivity (e.g., hooks, event listeners like `onClick`).
- **Suspense & Streaming**: Wrap heavy or async components in `<Suspense>`.
- **Static Site Generation (SSG)**: Landing pages are fully statically generated by default in the Next.js App Router for optimal load speeds.

## 3. Performance & Image Loading
- **Image Optimization**: Use the `next/image` component for all images.
  - Add `priority` to images above the fold (e.g., Hero image) to improve LCP (Largest Contentful Paint).
  - Use `sizes` for responsive layouts.
  - Implement `placeholder="blur"` and `blurDataURL` for progressive loading.
- **Bundle & Loading**:
  - Leverage `next/font` for zero layout shift.
  - Use `next/dynamic` to lazy load heavy client components below the fold.
  - Next.js automatically route-level code splits and prefetches links on hover.

## 4. SEO Layer
- **Metadata API**: Export standard `generateMetadata()` from `layout.js` or `page.js` to manage:
  - `title`, `description`
  - `openGraph` images and `twitter` card configurations
  - `alternates.canonical`
- **Structured Data**: Implement JSON-LD arrays where necessary:
  - Organization (`Organization`)
  - WebSite (`WebSite`)
  - FAQ Schema (`FAQPage` mapped in the FaqsSection)
- **Crawl Signals**: Use `sitemap.js` and `robots.js` (using Next.js `MetadataRoute`) to keep bots automatically synced.

## 5. Component Design & Styling
- **Atomic Design**: Structure flows sequentially (`ui` -> `sections` -> `page`).
- **Variants**: Use Class Variance Authority (`cva`) alongside `clsx` and `tailwind-merge` to handle UI states and component variations predictably (e.g., `Button` variants).
- **Styling**: Stick to Tailwind CSS utility classes combined with custom CSS variables (tokens) in `globals.css` for standardized themes.

## 6. Maintainable Code Standards
- **ESLint & Prettier**: Conform to `next/core-web-vitals` and maintain consistent code formatting.
- **Absolute Imports**: Always use absolute path aliases (e.g., `@/components/ui/button`) as defined in `jsconfig.json`/`tsconfig.json`.
- *(Note: While the ideal architecture incorporates strict TypeScript, Husky, Vitest, etc., ensure you align with the stack established in the repository e.g. using structured JavaScript).*
